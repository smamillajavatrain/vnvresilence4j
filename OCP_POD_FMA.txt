| Failure Mode           | How to Inject the Failure?                                               | Detectability                                                       | Impact to Application                                       | Response                                                                                   |
| ---------------------- | ------------------------------------------------------------------------ | ------------------------------------------------------------------- | ----------------------------------------------------------- | ------------------------------------------------------------------------------------------ |
| **Kill Main Process**  | Execute `kill 1` inside the pod using `kubectl exec`                     | Pod restart count increases, restart logs observed, alert triggered | Temporary request failures during restart, short downtime   | Kubernetes auto-restarts pod, traffic redirected to healthy pods, system recovers quickly  |
| **Force Kill Process** | Execute `kill -9 1` inside the pod                                       | Sudden error spike, pod restart alert, abrupt termination logs      | Immediate failures, no graceful shutdown, short error burst | Auto-restart by Kubernetes, retry mechanisms handle failures, traffic shifts to other pods |
| **Delete Pod**         | Run `kubectl delete pod <pod-name>`                                      | Pod termination logs, normal lifecycle events                       | Minimal or no impact due to graceful shutdown               | Requests drained properly, new pod created, zero or negligible downtime                    |
| **Force Delete Pod**   | Run `kubectl delete pod <pod-name> --grace-period=0 --force`             | Pod disappears instantly, error spike observed                      | Short-lived request failures, abrupt disruption             | Kubernetes recreates pod, traffic fails over, system stabilizes quickly                    |
| **CrashLoopBackOff**   | Introduce misconfiguration or failing dependency (e.g., wrong DB config) | High restart count, CrashLoopBackOff state, alerts triggered        | Continuous failures from affected pod, increased error rate | Requires manual fix, other pods continue serving traffic, alerts help detection            |
| **DB Blackhole**       | Block DB traffic using iptables (`DROP` DB port)                         | Timeout errors, latency spike, DB connection alerts                 | Requests hang and fail after timeout, degraded performance  | Retry logic and timeouts handle failures, system recovers after network restored           |
| **DB Packet Loss**     | Introduce packet loss using `tc netem loss 30%`                          | Increased latency, intermittent errors, monitoring alerts           | Partial failures, unstable responses, degraded performance  | Retries mitigate some failures, system stabilizes after network normalization              |
| **DB Latency**         | Inject delay using `tc netem delay 200ms`                                | High response time metrics, latency alerts                          | Slow responses, performance degradation                     | System continues functioning but slower, recovers immediately after latency removed        |
| **DB Shutdown**        | Stop or restart DB instance (RDS reboot / DB stop)                       | DB down alert, connection errors, error spike                       | Complete service disruption for DB-dependent operations     | Application reconnects after DB is up, retry and failover mechanisms restore service       |
